# Dialogue Parser & Dataset Generator (Lilith Project)

Этот проект предназначен для парсинга игровых диалогов в формате **Yarn**, построения графа связей между репликами, фильтрации диалоговых веток и генерации финальных датасетов в формате `.jsonl`.

## 📂 Обзор модулей

### 1. `fp.py` (File Parser)
**Назначение:** Низкоуровневый парсинг `.yarn` файлов. 
- Использует регулярные выражения для поиска ID строк (`#line:ID`), команд перехода (`<<jump>>`) и выборов игрока (`->`).
- Основная сложность здесь — обработка вложенных выборов на основе отступов.
- **Ключевые функции:**
    - `parse_file(filepath, graph)`: Читает файл построчно, отслеживает текущий узел (`title`), обрабатывает блоки выборов и создает узлы в графе.
    - `extract_line_id(line)`: Вырезает шестнадцатеричный ID строки.
    - `ChoiceBlock`: Вспомогательный класс для управления стеком вложенности выборов.

### 2. `graph.py`
**Назначение:** Структура данных графа диалогов.
- Определяет классы `DialogueNode` (узел) и `DialogueGraph` (весь граф).
- **Ключевые функции:**
    - `resolve_jumps()`: Связывает узлы, содержащие команду `jump`, с начальными узлами целевых заголовков (`title`).
    - `build_graph(folder)`: Обходит папку с файлами и запускает процесс парсинга.
    - `find_entry_node(title)`: Ищет «входную» ноду для конкретного диалогового блока.

### 3. `translate.py`
**Назначение:** Загрузка и поиск переводов.
- Работает с CSV-файлами, где сопоставляются ID строк и текст на разных языках (RU, EN, ZH).
- **Ключевые функции:**
    - `load_translation(path)`: Загружает переводы в словарь для быстрого доступа.
    - `id_to_text(node_id, translation)`: Возвращает текст по ID.

### 4. `text.py`
**Назначение:** Обработка и нормализация текста.
- Определяет, кто говорит фразу, и чистит текст от технических тегов игры.
- **Ключевые функции:**
    - `detect_speaker(text)`: Ищет маркеры говорящего (например, `Lilith:`).
    - `normalize_speaker(speaker)`: Приводит имена к единым тегам (`YOU`, `LILITH`, `OTHER`).
    - `clean_final_text(text)`: Удаляет системные команды в скобках (типа `[lock if...]`), маркеры успеха/провала и лишние пробелы.

### 5. `subgraphs.py`
**Назначение:** Группировка узлов и поиск путей.
- Разбивает огромный общий граф на «подграфы» (связные куски диалогов).
- **Ключевые функции:**
    - `find_valid_subgraphs(graph, translation)`: Находит компоненты связности, состоящие только из релевантных персонажей.
    - `extract_paths_from_subgraph(...)`: С помощью поиска в глубину (DFS) находит все возможные линейные пути (цепочки реплик) в подграфе.

### 6. `tokenizer.py`
**Назначение:** Контроль размера контекстного окна.
- Пытается использовать библиотеку `tiktoken` (cl100k_base), если она не установлена — использует эвристический fallback-метод для подсчета токенов.
- Помогает обрезать слишком длинные диалоги, чтобы они вписывались в лимиты LLM (например, 1024 токена).

### 7. `debug.py`
**Назначение:** Инструментарий для отладки.
- Позволяет «протрассировать» конкретную строку: найти её в графе, посмотреть, попала ли она в подграфы и почему была отфильтрована.
- **Ключевые функции:**
    - `visualize_node_branch(target_id, ...)`: Рисует в консоли путь «до» и «после» указанной ноды для проверки логики.

### 8. `lolmake.py` (Main Orchestrator)
**Назначение:** Главный скрипт сборки датасета.
- Связывает все модули воедино.
- Извлекает метаданные (эмоции персонажей и локации сценария).
- Формирует финальные `.jsonl` файлы с разметкой тегами (например, `<CHAR_Lilith>`, `<SCENE>`).

## ⚙️ Алгоритм работы

Процесс обработки данных выглядит следующим образом:

1.  **Сбор Графа:** Скрипт сканирует папку с `.yarn` файлами. Парсер (`fp.py`) строит связи, учитывая переходы (`jump`) и ветвления.
2.  **Загрузка Текста:** Загружается CSV с переводами. Теперь каждый узел графа знает свой текст на всех языках.
3.  **Выделение подграфов:** Граф фильтруется. Удаляются системные узлы и узлы сторонних NPC (если нужно). Оставшиеся узлы группируются в связные блоки диалогов.
4.  **Поиск путей:** Внутри каждого блока запускается DFS. Мы получаем список всех возможных «прохождений» диалога от начала до конца.
5.  **Обогащение метаданными:** Для каждого пути из текста вытягиваются команды смены эмоций (спрайтов) и фонов (`back_creat`). Эти данные превращаются в описательные теги (например, `*спокойно*`).
6.  **Токенизация и окна:** Каждый путь проверяется на длину. Если диалог длинный, он нарезается на «окна», не превышающие `MAX_TOKENS`.
7.  **Сохранение:** Чистый текст без системного мусора записывается в JSONL, раздельно для каждого языка.

## 🚀 Как использовать

1.  Положи свои `.yarn` файлы в папку, указанную в `YARN_FOLDER`.
2.  Убедись, что файл переводов `en_ru_pairs.csv` доступен.
3.  Запусти сборку:
    ```bash
    python lolmake.py
    ```
4.  Если какая-то ветка не попадает в датасет, используй `debug.py` для поиска причины:
    ```python
    from debug import debug_trace_line
    debug_trace_line("your_line_id", graph, subgraphs, real_dialogs, translation)
    ```
